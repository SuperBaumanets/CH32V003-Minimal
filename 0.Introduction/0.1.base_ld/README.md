## Проверка размещения секций в памяти CH32V003

README для отладки минимального скрипта линкера (`ld`). Необходим, чтобы убедиться, что код, константы, инициализированные и неинициализированные переменные попадают в правильные секции памяти.

### Структура секций в ELF-файле

| Секция      | Назначение                                                                                                 | Расположение в памяти |
|-------------|------------------------------------------------------------------------------------------------------------|------------------------|
| `.text`     | Исполняемый код (инструкции).                                                                              | Flash (`0x08000000`)   |
| `.rodata`   | Константные данные только для чтения.                                | Flash (после `.text`)  |
| `.data`     | Инициализированные глобальные и статические переменные. **Начальные значения хранятся во Flash**, при старте должны копируются в SRAM. | SRAM (`0x20000000`)    |
| `.bss`      | Неинициализированные глобальные и статические переменные. В ELF-файле не занимают места, но резервируют область в SRAM, которая обнуляется при старте. | SRAM (после `.data`)   |
| `.sdata` / `.sbss` | *Small data* секции, автоматически создаваемые компилятором для маленьких переменных при включённой оптимизации `-msmall-data-limit`. | SRAM (внутри `.data` / `.bss`) |

> **Важно**: В данном примере компилятор по умолчанию помещает переменные размером ≤ 8 байт в специальные «small data» секции (`.sdata`, `.sbss`, `.srodata`). Это видно в map-файле: переменные `data_word` и `bss_counter` оказались в `.sdata` и `.sbss` соответственно. Можно отключить флагом `-msmall-data-limit=0`.


### Анализ MAP-файла

На основе полученного map-файла (`build/firmware.map`) можно посмотреть, как линкер распределил переменные и код по памяти:

#### Секция `.text` (код во Flash)

| Символ       | Секция         | Адрес (VMA)   | Размер | Примечание                                         |
|--------------|----------------|---------------|--------|----------------------------------------------------|
| `main_loop`  | `.text`        | `0x08000000`  | 2 байта| Бесконечный цикл.|
| `main`       | `.text.startup`| `0x08000002`  | 2 байта| Точка входа, вызывает `main_loop`.                 |
| **Итого**    |                |               | **4 байта** | Секция выровнена до 4 байт (`. = ALIGN(4)`).      |

#### Секция `.rodata` (константы во Flash)

| Символ        | Секция    | Адрес (VMA)   | Размер | Значение (hex)                                |
|---------------|-----------|---------------|--------|-----------------------------------------------|
| `ro_values`   | `.rodata` | `0x08000004`  | 12 байт| `0x44444444`, `0x55555555`, `0x66666666`      |
| `ro_message`  | `.rodata` | `0x08000010`  | 16 байт| Строка `"This is .rodata"` + нуль-терминатор |
| **Итого**     |           |               | **28 байт** | Выровнено до 4 байт, следующая секция с `0x08000020`. |

#### Секция `.data` / `.sdata` (инициализированные переменные в SRAM)

Начальные значения хранятся во Flash (LMA), а во время выполнения переменные доступны по адресам в SRAM (VMA).

| Символ       | Секция   | VMA (SRAM)     | LMA (Flash)    | Размер | Начальное значение |
|--------------|----------|----------------|----------------|--------|--------------------|
| `data_buff`  | `.data`  | `0x20000000`   | `0x08000020`   | 12 байт| `{0x11111111, 0x22222222, 0x33333333}` |
| `data_word`  | `.sdata` | `0x2000000c`   | `0x0800002c`   | 4 байта| `0xFFFFFFFF`       |
| **Итого**    |          |                |                | **16 байт** | Выровнено до 4 байт. |

**Пояснение:**
- Переменная `data_word` оказалась в секции `.sdata`, а не `.data`, потому что её размер (4 байта) меньше порога small data (по умолчанию 8 байт). Массив `data_buff` превышает этот порог (12 байт), поэтому остался в обычной `.data`.
- LMA для `.data` вычислен автоматически директивой `> RAM AT > FLASH` и начинается сразу после `.rodata` (`0x08000020`).

#### Секция `.bss` / `.sbss` (неинициализированные переменные в SRAM)

Переменные из `.bss` не занимают места в ELF-файле, но резервируют память в SRAM. При старте они должны быть обнулены.

| Символ        | Секция   | VMA (SRAM)     | Размер | Примечание                                     |
|---------------|----------|----------------|--------|------------------------------------------------|
| `bss_area`    | `.bss`   | `0x20000010`   | 12 байт| Массив из 3 элементов `unsigned int`.          |
| `bss_counter` | `.sbss`  | `0x2000001c`   | 4 байта| Одиночная переменная, попала в small bss.      |
| **Итого**     |          |                | **16 байт** | Выровнено до 4 байт, конечный адрес `0x20000020`. |

**Пояснение:**
- Аналогично `.data`, одиночная переменная `bss_counter` помещена в `.sbss` из-за малого размера.


### Сборка прошивки
```bash
# Базовая сборка
make all
```

После успешной сборки в папке `build/` будут созданы файлы:
- `firmware.elf` - ELF файл для отладки
- `firmware.bin` - бинарный файл для прошивки
- `firmware.map` - MAP файл для отладки
